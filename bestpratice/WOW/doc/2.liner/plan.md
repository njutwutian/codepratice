# WOW 新迭代执行计划（高性能架构视角）

## 1. 目标与范围

基于 `doc/happypath` 已打通的链路（创建事件 -> 列表展示 -> 详情查看），本次迭代聚焦“可规模化”的事件链路能力，覆盖以下目标：

1. 后端吞吐能力：稳定支撑最高 1000 条/秒事件写入（持续压测窗口内达标）。
2. 前端性能体验：事件列表使用虚拟列表滚动加载，保证大数据量场景下不卡顿。
3. 查询筛选能力：后端支持按 `eventType`、`eventStatus` 条件筛选并分页返回。
4. 模拟器能力升级：`SecretPalace` 支持动态生成不同 `eventType`、`eventStatus` 的组合数据并可控发压。
5. 消息削峰能力：后端事件写入链路强制引入 Kafka 作为缓冲与异步处理通道。
6. 实时推送能力：后端提供 WebSocket 接口，向前端推送事件新增与状态变化。

本计划涉及子系统：

- `src/GoldenMount`（后端）
- `src/NorthToward`（前端）
- `src/SecretPalace`（模拟器）

---

## 2. 本次迭代 KPI（验收口径）

### 2.1 后端 KPI

1. 写入吞吐：`>= 1000 events/s`（10 分钟稳定窗口，平均值达标）。
2. 写入成功率：`>= 99.9%`。
3. 写入接口延迟：P95 `< 120ms`，P99 `< 250ms`（不含客户端网络抖动）。
4. 查询接口延迟（带筛选）：P95 `< 200ms`（分页 size=50）。

### 2.2 前端 KPI

1. 虚拟列表滚动帧率：主流开发机保持 `>= 55 FPS`。
2. 单页可视渲染：列表数据量达到 5 万条时，滚动无明显卡顿。
3. 筛选操作响应：切换 `eventType/eventStatus` 后首屏结果渲染 `< 400ms`（本地联调环境）。

### 2.3 模拟器 KPI

1. 支持按配置动态生成 `eventType`、`eventStatus` 组合数据。
2. 支持固定速率、阶梯升压、突发冲击三种发压模式。
3. 支持生成报告：总发送数、成功率、失败原因分布、每秒吞吐曲线。

## 3. 架构改造方案

## 3.1 GoldenMount（后端）

### 3.1.1 接口与查询模型升级

1. 保持现有 `POST /api/v1/events`。
2. 升级 `GET /api/v1/events`，新增查询参数：
	- `eventType`（可选）
	- `eventStatus`（可选）
	- `pageNo`（默认 1）
	- `pageSize`（默认 50，最大 200）
	- `sort`（默认 `eventTime,desc`）
3. 返回结构补充分页元信息：`total/pageNo/pageSize/hasNext`。

### 3.1.2 数据库与索引优化

1. 在 `gm_event_record` 增加复合索引：
	- `(event_type, event_status, event_time desc)`
	- `(created_at desc)`
2. 保留并使用 `request_id` 唯一约束保证幂等。
3. 评估时间分区（按日/周）策略，预留分区迁移脚本。
4. Flyway 增加增量迁移，避免手工改表。

### 3.1.3 写入性能优化

1. 批量写入路径：支持模拟器批次提交（可选 API：`POST /api/v1/events/batch`）。
2. 连接池参数调优（HikariCP）：
	- `maximumPoolSize` 按压测结果逐步提升。
	- 合理设置 `connectionTimeout`、`idleTimeout`、`maxLifetime`。
3. 减少非必要同步日志 IO，关键日志异步化。
4. 统一使用预编译 SQL，避免动态拼接导致执行计划不稳定。

### 3.1.4 可观测与限流保护

1. 增加指标：QPS、错误率、DB 慢查询、线程池队列长度。
2. 增加保护：
	- 网关/应用级限流（防止瞬时过载）。
	- 降级策略：高负载时限制超大 `pageSize`。
3. 压测期间保留性能快照：CPU、内存、GC、连接池占用。

### 3.1.5 Kafka 削峰与异步处理（强制）

1. 新增 Kafka Topic：`wow.events.raw`（事件入站）。
2. `POST /api/v1/events` 改为“写 Kafka + 快速确认”，由消费者异步落库。
3. 新增 Kafka Consumer 组：`goldenmount-event-writer`，负责批量消费并持久化。
4. 保证幂等：继续依赖 `request_id` 唯一约束，处理重复消息。
5. 新增失败处理：
	- 消费失败重试（有限次数）。
	- 超限后写入死信 Topic：`wow.events.dlq`。
6. 新增可观测指标：生产成功率、消费延迟、消费堆积、DLQ 数量。

### 3.1.6 WebSocket 实时推送（强制）

1. 新增 WebSocket 端点：`/ws/events`。
2. 推送消息类型：
	- `EVENT_CREATED`
	- `EVENT_STATUS_CHANGED`
3. 推送负载最小字段：`eventId`、`eventType`、`eventStatus`、`eventTime`、`createdAt`。
4. 与 Kafka 联动：消费者落库成功后触发 WebSocket 推送。
5. 支持连接管理：心跳、断线重连提示、服务端连接数监控。

## 3.2 NorthToward（前端）

### 3.2.1 虚拟列表方案

1. 在事件列表页引入虚拟滚动（如 react-window/react-virtualized 同类方案）。
2. 分离“数据页缓存”与“渲染窗口”：
	- 数据按页增量拉取。
	- 仅渲染可视区 + 预加载区。
3. 行高固定或可预测，避免频繁重排。

### 3.2.2 滚动与轮播不卡顿

1. 列表项组件 `memo` 化，避免全量重渲染。
2. 使用请求合并/防抖，避免滚动触底多次并发请求。
3. 图片/重组件延迟加载，避免主线程阻塞。
4. 筛选条件变化时执行“状态重置 + 首屏优先”渲染策略。

### 3.2.3 筛选交互

1. 页面提供 `eventType`、`eventStatus` 双筛选。
2. 筛选值通过 query 参数回传后端，不在前端做全量过滤。
3. 保留筛选状态到 URL，支持刷新恢复。

### 3.2.4 WebSocket 实时订阅

1. 新增 WebSocket 客户端连接 `ws/events`。
2. 接收 `EVENT_CREATED`、`EVENT_STATUS_CHANGED` 后更新 Dashboard 与列表缓存。
3. 对当前筛选条件做增量命中判断，避免全量刷新。
4. 断线自动重连，重连后触发一次增量拉取补偿。

## 3.3 SecretPalace（模拟器）

### 3.3.1 动态数据生成引擎

1. 基于配置生成事件：
	- `eventType` 候选集合及权重。
	- `eventStatus` 候选集合及权重。
	- 字段模板与随机规则。
2. 支持时间分布策略：均匀、泊松、突发。
3. 支持 requestId 生成策略，确保幂等键唯一。

### 3.3.2 发压控制

1. 固定速率：例如 `--rate 1000`。
2. 阶梯升压：如 `200 -> 500 -> 800 -> 1000`。
3. 突发冲击：短时高于 1000 的尖峰，验证系统保护策略。

### 3.3.3 报告与回放

1. 输出 JSON/Markdown 报告。
2. 保存失败样本用于回放。
3. 支持按 `eventType/eventStatus` 维度输出成功率分布。

## 5. 风险与应对

1. 风险：1000/s 下 MySQL 写入抖动明显。
	- 应对：批量写入、事务粒度优化、必要时引入消息队列缓冲层。
2. 风险：前端列表重渲染导致掉帧。
	- 应对：虚拟列表 + 组件 memo + 精简列表项结构。
3. 风险：筛选条件组合导致慢查询。
	- 应对：复合索引 + explain 分析 + 限制超大页。
4. 风险：模拟数据分布与真实场景偏差大。
	- 应对：引入权重配置与真实样本回放机制。
5. 风险：Kafka 消费堆积导致数据展示延迟。
	- 应对：增加消费并发、批量提交优化、堆积告警阈值。
6. 风险：WebSocket 连接激增导致推送延迟或断连。
	- 应对：连接数上限、心跳保活、广播分片与降频策略。

## 8. 本轮联调结论沉淀（2026-06）

### 8.1 WebSocket 推送模型结论

1. 推送模式从“每事件一条 detail”切换为“每秒批量一次 stats 快照”。
2. WebSocket 消息以聚合结果为主，推荐统一使用 `EVENT_STATS` 作为前端消费主类型。
3. 同一端点 `ws/events` 可继续沿用，不强制新增第二个 WebSocket 端点。
4. 保留手动触发快照推送能力，作为重连校准与排障手段。

### 8.2 前端消费模式结论

1. Dashboard 采用“读写分离”策略：
	- WebSocket 回调只写入内存变量（缓存最新快照）。
	- UI 每 2 秒统一读取缓存并 `setState`。
2. 该策略可降低高频消息下主线程抖动，避免每条消息触发渲染。
3. 首屏与筛选场景可保留 HTTP 兜底，避免过滤维度与全量快照语义冲突。

### 8.3 性能瓶颈结论（面向 1000 QPS）

1. 仅降低前端拉取频率不足以达成 1000 QPS，核心瓶颈仍在后端写入与聚合查询路径。
2. 当前写入链路存在“查重 + 插入”双往返，建议改为“直接插入 + 唯一键异常处理”。
3. 聚合维度缺少关键复合索引（`event_type`、`event_status`、时间维度），在数据量增长后会显著放大查询成本。
4. 每秒全表聚合属于过渡方案，长期建议演进为增量统计（内存计数/统计表）以降低持续压测成本。

### 8.4 与既有计划的对齐

1. 本轮结论与“Kafka 削峰 + WebSocket 实时推送”的主方向一致。
2. 在阶段 B/E 中补充以下优先项：
	- 写入路径去除前置查重。
	- 新增聚合复合索引 Flyway 迁移。
	- WebSocket 快照协议文档化（字段、频率、重连校准策略）。
3. 在阶段 C 中补充“前端读写分离消费模型”的验收检查点。

### 8.5 下一步落地优先级（建议）

1. P0：固化 WebSocket `EVENT_STATS` 契约并更新前后端联调文档。
2. P0：补齐 `gm_event_record` 聚合相关复合索引与压测回归。
3. P1：写入接口改造为“直接写入 + 唯一键兜底”，减少 DB 往返。
4. P1：前端在筛选模式下定义“HTTP 兜底 or 订阅分流”的统一策略。
5. P2：评估增量统计方案，替代全表聚合。
